# Проектирование Data Pipeline для системы рекомендаций

## 1. Источники данных (Data Sources)

Система рекомендаций получает данные из четырёх основных источников:

| Источник | Тип загрузки | Частота | Протокол |
|----------|-------------|---------|----------|
| Клики и просмотры пользователей | **Streaming** | Real-time | WebSocket / JS SDK → Kafka |
| Добавления в корзину и покупки | **Streaming** | Real-time | WebSocket / JS SDK → Kafka |
| Каталог товаров (ERP) | **Batch** | Ежедневно (02:00 UTC) | JDBC / REST API → Airflow |
| Профили клиентов (CRM) | **Batch** | Ежедневно (03:00 UTC) | REST API → Airflow |

**Обоснование разделения:** поведенческие данные пользователей (клики, просмотры, корзина) требуют обработки в реальном времени для актуальных рекомендаций. Каталог товаров и профили клиентов обновляются реже и допускают батчевую обработку.

## 2. Архитектура Pipeline (ETL/ELT Design)

Архитектура построена по принципам **Kappa Architecture** с дополнительным батчевым слоем для каталожных данных. Все стриминговые данные проходят через единый слой обработки (Flink), что устраняет необходимость дублирования логики между потоковой и пакетной обработкой.

### 2.1 Streaming Pipeline (пользовательское поведение)

```
Клики/Корзина → Kafka → Flink → Great Expectations → S3 (raw → cleaned) → Feast
```

**Этап очистки (Flink):**
- Дедупликация событий по `event_id` (exactly-once semantics через Kafka transactional API)
- Валидация схемы (JSON Schema) — отбраковка событий с некорректными полями
- Обогащение данных: добавление `session_id`, геолокации по IP, категории устройства
- Оконная агрегация (tumbling window 5 мин): подсчёт кликов, среднее время просмотра, CTR по категориям

### 2.2 Batch Pipeline (каталог товаров)

```
ERP → Airflow → Spark → Great Expectations → S3 (raw → cleaned) → Embedding Service → Vector DB
```

**Этап очистки (Spark):**
- Дедупликация товаров по `product_id` с приоритетом последнего обновления
- Нормализация текстовых атрибутов (lowercase, удаление HTML, стандартизация единиц измерения)
- Заполнение пропущенных значений (median для числовых, "unknown" для категориальных)
- Валидация бизнес-правил: цена > 0, наличие обязательных полей (название, категория)

### 2.3 Генерация эмбеддингов

Эмбеддинги генерируются на **двух этапах**:

1. **Item-эмбеддинги** (Batch, после обработки каталога в Spark):
   - Модель: SentenceTransformers (`all-MiniLM-L6-v2`) для текстовых описаний
   - Вход: название + описание + категория товара
   - Выход: 384-мерный вектор → записывается в Vector DB
   - Частота: ежедневно после обновления каталога

2. **User-эмбеддинги** (Near-real-time, на основе сессий из Flink):
   - Модель: обученная двухбашенная модель (Two-Tower) на исторических данных
   - Вход: последовательность взаимодействий пользователя за сессию
   - Выход: 384-мерный вектор → обновляется в Vector DB и Online Store
   - Частота: каждые 5 минут (по завершении окна агрегации)

## 3. Выбор хранилищ (Storage Selection)

| Хранилище | Технология | Назначение | Обоснование |
|-----------|-----------|------------|-------------|
| **Message Broker** | Apache Kafka | Приём стриминговых событий | Высокая пропускная способность (миллионы событий/сек), гарантия доставки, replay capability для повторной обработки |
| **Data Lake** | Amazon S3 / MinIO | Хранение сырых и очищенных данных (Parquet) | Дешёвое масштабируемое хранение, поддержка партиционирования по дате, совместимость с Spark/Flink |
| **Feature Store** | Feast | Управление фичами для ML | Единый источник фичей для обучения и инференса, предотвращение Training-Serving Skew (см. раздел 4) |
| **Offline Store** | PostgreSQL / Redshift | Хранение исторических фичей | Поддержка point-in-time joins для корректного обучения, SQL-интерфейс для Data Scientists |
| **Online Store** | Redis | Быстрый доступ к фичам при инференсе | Латентность < 1 мс, TTL для автоматической инвалидации устаревших фичей |
| **Vector DB** | Pinecone / Qdrant | Хранение и поиск эмбеддингов | ANN-поиск (HNSW) с латентностью < 50 мс, поддержка фильтрации по метаданным, горизонтальное масштабирование |
| **Metadata DB** | PostgreSQL | Каталог товаров, метаданные | ACID-транзакции, зрелая экосистема, Foreign Keys для referential integrity |
| **Model Registry** | MLflow | Версионирование моделей | Tracking экспериментов, staging/production promotion, артефакты моделей |

### Обоснование ключевого стека: Kafka → Flink → S3 → Feast → Pinecone

- **Kafka → Flink** (вместо Kafka → Spark Streaming): Flink обеспечивает настоящую потоковую обработку (event-at-a-time), а не микро-батчи, что даёт меньшую латентность для пользовательских событий.
- **S3 как Data Lake**: разделение вычислений и хранения позволяет независимо масштабировать каждый слой. Формат Parquet обеспечивает эффективное колоночное хранение.
- **Feast как Feature Store**: открытый стандарт, интеграция с offline (PostgreSQL) и online (Redis) хранилищами, поддержка point-in-time correctness.
- **Pinecone/Qdrant**: управляемый сервис для векторного поиска, не требует операционных затрат на поддержку индексов.

## 4. Data Governance: консистентность между обучением и инференсом

### 4.1 Проблема Training-Serving Skew

Training-Serving Skew возникает, когда фичи, используемые при обучении модели, вычисляются иначе, чем при инференсе. Это приводит к деградации качества модели в production.

### 4.2 Стратегия предотвращения через Feature Store (Feast)

**Единый источник трансформаций:**
- Все фичи определяются один раз в Feast Feature Repository (Python-декларации).
- Одни и те же определения фичей используются и для наполнения Offline Store (обучение), и для материализации в Online Store (инференс).
- Это гарантирует, что трансформации идентичны в обоих путях.

**Point-in-time correctness для обучения:**
- При создании обучающего датасета Feast выполняет point-in-time join: для каждого примера берутся значения фичей, актуальные на момент события (а не текущие значения). Это предотвращает data leakage.

**Материализация в Online Store:**
- Feast периодически (каждые 5 минут) материализует свежие фичи из Offline Store в Redis (Online Store).
- При инференсе API-сервис запрашивает фичи из Redis по `user_id` / `item_id`.
- TTL на ключах Redis = 24 часа, что предотвращает использование устаревших данных.

### 4.3 Мониторинг консистентности

| Метрика | Инструмент | Порог алерта |
|---------|-----------|--------------|
| Feature freshness (задержка материализации) | Prometheus + Grafana | > 15 минут |
| Feature distribution drift | Great Expectations | KL-divergence > 0.1 |
| Schema validation errors | Great Expectations | > 0.1% от батча |
| Missing feature rate при инференсе | Prometheus | > 1% запросов |
| Model prediction drift | MLflow + Evidently | PSI > 0.2 |

### 4.4 Отказоустойчивость

- **Kafka**: replication factor = 3, min.insync.replicas = 2 — защита от потери сообщений при отказе брокера.
- **Flink**: включены чекпоинты (interval = 60 сек, exactly-once), при сбое — автоматический рестарт с последнего чекпоинта.
- **Airflow**: retry policy (3 попытки с exponential backoff), алерты в Slack при провале DAG.
- **Feature Store**: fallback на Offline Store при недоступности Redis; кэширование последних известных значений фичей на стороне API.
- **Dead Letter Queue (Kafka DLQ)**: события, не прошедшие валидацию, направляются в отдельный топик для последующего анализа без блокировки основного потока.

### 4.5 Информационная безопасность

- Данные в S3 шифруются at-rest (AES-256) и in-transit (TLS 1.3).
- Доступ к Kafka и Feature Store через mTLS и RBAC.
- PII-данные (email, имя) маскируются на этапе очистки в Flink; в Feature Store хранятся только анонимизированные идентификаторы.
- Audit log всех обращений к данным через Apache Atlas (Data Lineage).
